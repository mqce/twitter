# ロボット対話インスタレーション 設計書

## 1. 目的

本設計書は、2台のロボットが対面状態になったことを赤外線通信で検知し、交互の会話を成立させるための実装方針を定義する。  
本システムにおける「会話」とは、各ロボットが **スピーカーで音を発すること** を指す。  
マイクで相手の音声を聞いて会話順序を判断するのではなく、赤外線で話者の交代タイミングを通知し合う。  
本システムでは、実装と保守の単純さを優先し、対面判定および会話進行制御は **赤外線通信のみ** で実現する。  
BLE は本設計には採用しない。

## 2. 設計方針

- 使用マイコンは `ESP32-C3 SuperMini` とする。
- 各ロボットは固定のペア相手を持ち、個体 ID を事前設定する。
- 対面検知は、指向性を持たせた赤外線送受信により行う。
- 対面判定は厳密な角度・距離測定ではなく、「一定時間、相互に赤外線が届いていること」を条件とする。
- 会話は必ず `Parent` から開始する。
- `Parent` が話し終わったことを赤外線で `Child` に通知し、`Child` はそれを受けて話し始める。
- `Child` が話し終わったら、同様に `Parent` に通知し、これを繰り返す。
- 状態数を減らすため、独立した `HANDSHAKE` 状態は持たず、`IDLE` 状態内の連続成功カウンタで対面成立を判定する。
- 会話中の状態は「どちらのターンか」を表す `PARENT_TALKING` と `CHILD_TALKING` を用いる。

## 3. システム構成

### 3.1 ハードウェア構成

各ロボットは以下の構成を持つ。

- マイコン: `ESP32-C3 SuperMini`
- 赤外線送信素子: `OSI5LA5113A`
- 赤外線受信素子: `TSSP4038`
- スピーカー

### 3.2 赤外線構成の前提

- 送信は 38kHz 変調で行う。
- `TSSP4038` は 38kHz の変調赤外線を検出する受信モジュールとして使用する。
- 送信 LED と受信モジュールには遮光フードまたは筒を設け、正面方向の指向性を強める。
- 自機送信の回り込みを避けるため、送信部と受信部は物理的に少し離して配置する。

## 4. 採用理由

### 4.1 赤外線のみを採用する理由

本案件で必要なのは以下である。

- 対面した相手にのみ届きやすい検知手段
- 固定ペア相手の識別
- 会話開始と話者交代の簡易な相互確認
- 通信断時の待機状態への復帰

これらは赤外線のみで実現可能であり、BLE を追加すると以下の複雑さが増える。

- アドバタイズ、スキャン、接続の状態管理
- 切断および再接続処理
- BLE と IR の二重状態管理
- 対面判定そのものは結局 IR に依存する設計になる

したがって、本システムでは BLE を併用せず、赤外線のみを採用する。

## 5. 論理構成

### 5.1 役割

各ペア内で 2 台の役割を固定する。

- `Parent`: 呼びかけを開始し、最初に話し始める側
- `Child`: 呼びかけに応答し、`Parent` の終了通知を受けて話し始める側

役割を固定することで、同時送信による衝突を避け、状態機械を単純化する。

### 5.2 個体識別

- 各ロボットは `self_id` と `partner_id` を保持する。
- 受信したフレーム内の送信元 ID を確認し、`partner_id` と一致した場合のみ有効な通信として扱う。
- 一致しない ID、または不正なフレームは破棄する。

### 5.3 会話進行モデル

- 会話順序は赤外線で管理する。
- 現在の話者は、音声再生中に定期的な生存通知を送る。
- 話者は音声再生終了時に終了通知を送る。
- 次の話者はその終了通知を受けて音声再生を開始する。
- 音声そのものはローカルに保持したデータをスピーカーから再生する。

## 6. 通信仕様

### 6.1 基本方式

赤外線通信は、`IDLE` 中の対面確認と、会話中のターン通知の2段階で構成する。

#### 対面確認

1. `Parent` が一定周期で `HELLO` を送信する
2. `Child` が受信し、相手 ID を確認する
3. `Child` が `ACK` を返す
4. `Parent` が `ACK` を受信したら連続成功回数を加算する
5. 規定回数の連続成功で対面成立とみなし、`Parent` が会話を開始する

#### 会話進行

1. `Parent` が話している間、`Parent` は定期的に「自分のターン継続中」であることを送信する
2. `Parent` が話し終わったら「Parent のターン終了」を送信する
3. `Child` はその通知を受けて話し始める
4. `Child` が話している間、`Child` は定期的に「自分のターン継続中」であることを送信する
5. `Child` が話し終わったら「Child のターン終了」を送信する
6. `Parent` はその通知を受けて再び話し始める

### 6.2 フレーム種別

送信するフレームは以下の 4 種類とする。

- `HELLO`
  - `Parent` が `IDLE` 中に送る呼びかけフレーム
- `ACK`
  - `Child` が `HELLO` に対して返す応答フレーム
- `ACTIVE`
  - 現在の話者が「まだ自分のターン中である」ことを示すフレーム
- `DONE`
  - 現在の話者が「話し終わった」ことを示すフレーム

### 6.3 ターン種別

`ACTIVE` と `DONE` は、どちらのターンに関する通知かを識別できる必要がある。  
そのため、フレームに `turn_owner` を持たせる。

- `P`
  - `Parent` のターン
- `C`
  - `Child` のターン

例:

- `ACTIVE,P,...`
- `DONE,P,...`
- `ACTIVE,C,...`
- `DONE,C,...`

### 6.4 フレーム内容

各フレームは、最低限以下の情報を持つ。

- 種別
- ターン種別
- 送信元 ID
- 宛先 ID

`HELLO` と `ACK` では `turn_owner` を空または予約値としてよい。  
実装方法はライブラリや送信方式に合わせて決めてよいが、可読性と実装容易性を優先し、短い固定長または短い区切り文字形式を推奨する。

例:

```text
[TYPE][TURN_OWNER][SRC_ID][DST_ID]
```

例:

- `HELLO,-,01,02`
- `ACK,-,02,01`
- `ACTIVE,P,01,02`
- `DONE,P,01,02`
- `ACTIVE,C,02,01`
- `DONE,C,02,01`

### 6.5 推奨タイミング

実装初期値として、以下の値を推奨する。

- `Parent` の `HELLO` 周期: 100ms
- `Child` の `ACK` 応答遅延: 20ms から 30ms
- 対面成立条件: 3 回連続成功
- `ACTIVE` 送信周期: 100ms から 200ms
- `DONE` 送信回数: 2 回から 3 回連続送信
- 通信断による待機復帰: 2000ms 無通信

これらの値は現場での光学条件と音声長に応じて調整可能とする。

## 7. 状態遷移

### 7.1 状態一覧

各ロボットは以下の状態を持つ。

- `IDLE`
  - 待機状態
- `PARENT_TALKING`
  - `Parent` のターン状態
- `CHILD_TALKING`
  - `Child` のターン状態

### 7.2 `HANDSHAKE` 状態を設けない理由

本設計では、対面成立のための試行中であっても専用の `HANDSHAKE` 状態には遷移しない。  
代わりに `IDLE` の内部で以下を管理する。

- 直近の有効受信時刻
- 連続成功回数
- 規定回数に達したかどうか

これにより、状態数を増やさずに対面成立判定を実装できる。

### 7.3 Parent の状態遷移

#### `IDLE`

- 一定周期で `HELLO` を送信する
- `partner_id` からの有効な `ACK` を受信したら連続成功回数を加算する
- 連続成功回数が閾値に達したら、自機の音声再生を開始し `PARENT_TALKING` へ遷移する
- 音声再生開始直後から `ACTIVE,P` を周期送信する
- 一定時間 `ACK` が得られなければ連続成功回数をリセットする

#### `PARENT_TALKING`

- 自機スピーカーで音声を再生する
- 再生中は `ACTIVE,P` を周期送信する
- 再生終了時に `DONE,P` を 2 回から 3 回送信する
- `DONE,P` の送信後、`CHILD_TALKING` へ遷移する
- 一定時間有効な相手通信が得られなければ `IDLE` に戻る

#### `CHILD_TALKING`

- `Child` からの `ACTIVE,C` を受信している間は待機する
- `Child` からの `DONE,C` を受信したら、自機の音声再生を開始し `PARENT_TALKING` へ遷移する
- 再生開始直後から `ACTIVE,P` を周期送信する
- 一定時間有効な `ACTIVE,C` または `DONE,C` が得られなければ `IDLE` に戻る

### 7.4 Child の状態遷移

#### `IDLE`

- `HELLO` の受信待ちを行う
- `partner_id` からの有効な `HELLO` を受信したら `ACK` を返す
- `Parent` からの `ACTIVE,P` を受信したら `PARENT_TALKING` へ遷移する
- 一定時間 `HELLO` が途切れた場合は内部カウンタをリセットする

#### `PARENT_TALKING`

- `Parent` の音声中は自機では音を出さない
- `ACTIVE,P` を受信している間は `Parent` のターン継続中とみなす
- `DONE,P` を受信したら、自機の音声再生を開始し `CHILD_TALKING` へ遷移する
- 再生開始直後から `ACTIVE,C` を周期送信する
- 一定時間有効な `ACTIVE,P` または `DONE,P` が得られなければ `IDLE` に戻る

#### `CHILD_TALKING`

- 自機スピーカーで音声を再生する
- 再生中は `ACTIVE,C` を周期送信する
- 再生終了時に `DONE,C` を 2 回から 3 回送信する
- `DONE,C` の送信後、`PARENT_TALKING` へ遷移する
- 一定時間有効な相手通信が得られなければ `IDLE` に戻る

## 8. 成立判定

### 8.1 対面成立条件

対面成立は、`IDLE` 状態で以下を満たしたときとする。

- 受信相手が `partner_id` と一致する
- `HELLO` と `ACK` の組み合わせが期待どおりである
- 連続して規定回数の `HELLO` / `ACK` 成功が確認できる

### 8.2 会話開始条件

会話開始は、以下の流れで成立させる。

1. `Parent` が `HELLO` / `ACK` の連続成功で対面成立を確認する
2. `Parent` が自機音声を再生開始する
3. `Parent` が `ACTIVE,P` を送信する
4. `Child` が `ACTIVE,P` を受信し、`PARENT_TALKING` に入る

これにより、両者が同じ会話セッションに入ったことを相互に確認できる。

### 8.3 厳密な角度・距離判定を行わない理由

本システムは、対面状態を高精度に測定することが目的ではない。  
鑑賞者がロボットを向かい合わせた結果として、赤外線の通信が安定して成立することを確認できれば十分とする。  
このため、角度センサや距離推定アルゴリズムは採用しない。

## 9. 復帰設計

- `PARENT_TALKING` または `CHILD_TALKING` の最中に、一定時間有効な赤外線通信が成立しなくなった場合は `IDLE` に戻る
- 復帰条件は単純化のため、`最後に有効な相手通信を受信してからの経過時間` のみで判定する
- `ACTIVE` が届かない、または `DONE` が受信できない場合は通信断とみなす
- 受信断が短時間で回復した場合のチャタリングを避けるため、復帰タイムアウトは `ACTIVE` 周期より十分長く取る

## 10. 実装上の注意

### 10.1 光学的な注意

- 送信 LED が広角すぎると、対面していない状態でも通信が成立する可能性がある
- 周囲の反射で誤検知する可能性がある
- 会場照明条件によって受信安定性が変化する可能性がある

対策として、以下を推奨する。

- 遮光フードを追加する
- 送受信部の向きを正面に絞る
- 1 回だけの受信では成立扱いにしない

### 10.2 ソフトウェア実装上の注意

- 赤外線送信は `ESP32-C3` の 38kHz 出力が安定して行える手段を使う
- 受信割り込みまたはポーリングのどちらを用いる場合でも、フレーム識別処理は短く保つ
- 状態遷移はタイマベースで管理し、ブロッキング待ちを避ける
- 音声再生処理と通信処理は分離し、再生中も `ACTIVE` を送れるようにする
- `DONE` は取りこぼし対策として単発ではなく複数回送信する
- 状態は「自分が話しているか」ではなく「今どちらのターンか」で持つと実装が単純になる

## 11. 実装優先度

初期実装は以下の順で進める。

1. 38kHz 赤外線送信の確認
2. `TSSP4038` での受信確認
3. `HELLO` / `ACK` の最小通信実装
4. 個体 ID 判定の追加
5. `IDLE` 内の連続成功回数による対面成立判定の追加
6. 音声再生中の `ACTIVE` 送信実装
7. 音声終了時の `DONE` 送信実装
8. `PARENT_TALKING` / `CHILD_TALKING` の状態遷移実装
9. 無通信タイムアウトによる `IDLE` 復帰の追加

## 12. 今後の拡張余地

本設計では BLE を採用しないが、将来的に以下が必要になった場合は別途検討する。

- 会話コンテンツや設定値の無線更新
- 複数ペア管理
- ログ収集や遠隔保守
- 赤外線成立後の補助チャネルとしての追加通信

現時点では、要件と実装優先度の観点から対象外とする。
